# Object-Oriented Programming (OOP)

## 1. Encapsulation (Data Hiding & Protection)

* **Definition:** The practice of bundling an object's data (attributes) and methods together into a single unit (a class), while explicitly restricting direct external access to sensitive internal states by using **private underscores** (`_` or `__`).
* **Analogy:** Think of an automated bank teller machine (ATM). Your actual cash balance is hidden deep inside a secure vault. You cannot reach your hands through the screen to manually add or subtract dollar bills. Instead, you must interface safely through public buttons (**getter and setter methods**) like `withdraw()` or `check_balance()`, which validate your security PIN first.

```python
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        # A single leading underscore '_' warns developers it's "protected" (internal)
        # A double underscore '__' triggers Name Mangling, strictly locking it from direct external access
        self.__balance = balance 

    # GETTER: A safe public method to view private data
    def get_balance(self):
        return self.__balance

    # SETTER: A managed public method to modify private data securely
    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount
            print(f"Deposited ${amount} successfully.")
        else:
            print("Invalid deposit amount!")

# Interfacing with the encapsulated object
account = BankAccount("Alice", 1000)

# Direct access is blocked by Python's engine!
# print(account.__balance) -> Raises an AttributeError!

account.deposit(500)
print(f"Current Balance: ${account.get_balance()}") # Output: Current Balance: $1500

```

## 2. Inheritance (Code Reuse & Hierarchy)

* **Definition:** A mechanism where a new child class (**subclass**) automatically adopts all the attributes and methods of an existing parent class (**superclass**), allowing you to reuse code and build specialized variations without rewriting everything from scratch.
* **Analogy:** Think of genetic inheritance. A child inherits basic biological features like eye color and blood type from their parents. However, the child can still grow up to learn unique individual skills (like playing the violin) that their parents never knew how to do.

```python
# The Parent Class (Superclass)
class Vehicle:
    def __init__(self, brand, color):
        self.brand = brand
        self.color = color

    def ignite_engine(self):
        print(f"The {self.brand} engine rumbles to life.")

# The Child Class (Subclass) inherits from Vehicle
class ElectricCar(Vehicle):
    # Using super().__init__ to let the parent handle core setup
    def __init__(self, brand, color, battery_capacity):
        super().__init__(brand, color) 
        self.battery_capacity = battery_capacity # Specialized unique child attribute

    # Specialized unique child method
    def charge_battery(self):
        print(f"Charging the {self.brand}'s {self.battery_capacity}kWh battery.")

# Creating the child instance
tesla = ElectricCar("Tesla", "Red", 75)

# Adopting parent behavior cleanly
tesla.ignite_engine()  # Output: The Tesla engine rumbles to life.
# Executing custom child behavior
tesla.charge_battery()  # Output: Charging the Tesla's 75kWh battery.

```

## 3. Polymorphism (Many Forms, One Interface)

* **Definition:** The ability of different object classes to share the exact same method interface name, but execute completely different internal actions under the hood based on their specific type behavior.
* **Analogy:** Imagine a universal remote controller with a giant button labeled `PLAY`. If you aim it at a DVD player and press play, it spins a physical disc. If you aim it at a streaming app and press play, it streams a digital video file. One unified command interaction layout triggers completely different mechanical execution rules.

```python
class Dog:
    def make_sound(self):
        return "Woof woof!"

class Cat:
    def make_sound(self):
        return "Meow!"

# A collection containing entirely different object types
pets = [Dog(), Cat()]

# Polymorphism allows us to cycle through them using the exact same method call name safely!
for pet in pets:
    print(pet.make_sound())

# Output:
# Woof woof!
# Meow!

```

## 4. Abstraction (Hiding Complexity)

* **Definition:** A structural blueprint enforcement technique that hides messy background implementation details from the user, forcing child sub-classes to establish specific, tailored structural rules by subclassing the base **Abstract Base Class (ABC)** interface module.
* **Analogy:** Think of driving a modern car. To slow down, you press your foot against the brake pedal. You don't need to understand hydraulic pressure calculations, brake fluid friction physics, or master brake pad clamping measurements. The brake pedal interface isolates and completely abstracts away that mechanical complexity from you.

```python
# Importing Python's Abstract Base Class infrastructure tools
from abc import ABC, abstractmethod

# The un-instantiable abstract structural blueprint
class TalkingToy(ABC):
    def __init__(self, name):
        self.name = name

    # Forcing all child classes to define this action or crash on startup
    @abstractmethod
    def speak(self):
        pass

# Specialized subclass implementations
class RobotToy(TalkingToy):
    def speak(self):
        print(f"{self.name} says beep boop!")

class DinosaurToy(TalkingToy):
    def speak(self):
        print(f"{self.name} says ROOOOAR!")

# Instantiating the clean, abstractly unified child interfaces
rusty = RobotToy("Rusty")
rex = DinosaurToy("Rex")

rusty.speak() # Output: Rusty says beep boop!
rex.speak()   # Output: Rex says ROOOOAR!

```
